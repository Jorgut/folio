# UX Laws · 用户体验法则

> 人机交互研究中被反复验证的规律。它们不是"建议"——在 44px 以下点击就是更容易点错，7 项以上的导航就是更难选。这些原则直接转换成 Folio 中的布局和交互规则。

---

## 1. Fitts's Law

**公式：** `Time = a + b × log₂(D / W + 1)`
- D = 到目标的距离
- W = 目标宽度

**核心结论：** 目标越大 + 越靠近 = 点击越快。屏幕边缘和角落天生是"无限大"的目标（因为鼠标/手指到边缘就停住了）。

**在 Folio 中的应用：**

```css
/* 所有可交互元素触控区 ≥ 44×44px */
.nav-dot::after {
  content: '';
  position: absolute;
  inset: calc((44px - 100%) / -2);
}
```

- 翻页箭头放在屏幕左右边缘（利用边缘的"无限大"优势）
- 导航圆点用不可见 `::after` 扩展到 44px，视觉上只有 12px 但触控面积 44px
- 关闭按钮放右上角（屏幕角落最高效）

---

## 2. Hick's Law

**公式：** `Time = a + b × log₂(n)`
- n = 选择数量

**核心结论：** 选择越少，决策越快。公式是指数级的——8 个选项不是 4 个的 2 倍时间，而是更多。

**在 Folio 中的应用：**

```
导航层级 ≤ 3 层（主导航 → 子导航 → 内容）
菜单项 ≤ 7 项
卡片每页 4-6 张（不是 12 张）
```

从"展示所有选项"到"展示最相关的选项"：
- 16 种布局不要求用户全看完——用决策矩阵直接推荐 4-6 种
- Tab 导航 ≤ 5 项，超过就用下拉/折叠
- Stats 布局每页 3-4 个数字（超过 4 个就失去冲击力）

---

## 3. Miller's Law

**公式：** 工作记忆容量 ≈ 7 ± 2 组块

**核心结论：** 人们一次只能记住 5-9 个信息组块。但"组块"是可以训练的——一个组块可以是一个字母，也可以是一个完整的句子。

**在 Folio 中的应用：**

| 布局 | 每页内容量 | 依据 |
|------|-----------|------|
| List | 4-6 项 | 7±2 的上限内留安全空间 |
| Gallery | 4-8 张 | 同上，图片比文字省认知 |
| Tab 导航 | ≤ 5 项 | 严格限制 |
| Stats | 3-4 个数字 | 每个数字需要单独关注 |

- 信息量超了 → 用分类/分层组织（创建一个组块 = 一个分类标题）
- 不是强制限制，是防止信息过载的指导线

---

## 4. Jakob's Law

**原理：** 用户期望在网站上使用与"其他网站"相同的交互模式。

**为什么：** 用户把之前在别的产品上学到的操作习惯带到你的产品里。如果你的翻页方向跟用户习惯的不一样，用户会觉得"不好用"。

**在 Folio 中的应用：**

- 翻页方向符合直觉：左/上 = 上一页，右/下 = 下一页
- 快捷键符合常见映射：Space 翻页、F 全屏、Esc 退出
- 导航点位置：底部居中——99% 的 presentation 都是这么做的
- 不要发明新的交互模式——用户不需要学习怎么用你的 deck

---

## 5. Aesthetic-Usability Effect

**原理：** 美观的设计被用户感知为更好用。这不是说好看 = 好用，而是好看的设计让用户更宽容——愿意容忍一些小问题。

**为什么：** 情感因素影响可用性判断。一个排版精致的 deck，即使内容有瑕疵，观众也会更信任。

**在 Folio 中的应用：**

- 精致排版（字距、行距、对齐）提升内容信任度——内容不变，排版变了，感受完全不同
- 统一主题色 vs 随意配色 → 用户感知的专业度差几个量级
- 注意：美观不能替代可用——不要为了好看牺牲 44px 触控目标或 4.5:1 对比度

---

## 总结：在 Folio 中的设计检查清单

- [ ] Fitts's Law：所有可交互元素触控区 ≥ 44px 吗？
- [ ] Hick's Law：导航层级 ≤ 3 层吗？同一层次的选择 ≤ 7 项吗？
- [ ] Miller's Law：每页信息量在 7±2 组块范围内吗？
- [ ] Jakob's Law：交互模式符合用户在其他产品中的习惯吗？
- [ ] Aesthetic-Usability：deck 的视觉统一性能让用户更信任内容吗？
